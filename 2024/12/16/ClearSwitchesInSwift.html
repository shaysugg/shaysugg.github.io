Clear Breaking in Swift

break is an essential and widely used keyword across many programming languages. In Swift, it serves the same purposes, However in switch statements unlike in C, it is not mandatory to include break in each case if the case contains at least one executable line.

let number = (0...10).randomElement()
  switch number {
  case 0:
    print("Zero") //no break needed
  case 1:
    print("One") //no break needed
  default:
    break //break needed
}

This auto break can result in more concise and readable code. It may also help prevent bug such as the crash of the AT&T network in 1990to happen. When programmers may miss a break or put break in wrong places, leading to the execution of unintended cases.

Since break can be used in other places especially loops, things can become a bit unclear in Swift when dealing with nested switch statements inside loops. Consider following code which demonstrate a game handling loop;

enum GameState { case paused, terminated, running}
var gameState = GameState.running
    
//game rendering loop
while true {
 switch gameState {
 case .terminated:
   handleTerminated()
   break // <- are we exiting the loop?
 case .paused:
   handlePause()
 case .running:
   handleRunning()
  }
}

This may not cause confusion in other languages like C because each case has it own break in order to function correctly, but here in Swift it’s not clear wether we are breaking from the parent loop or if the break is part of the switch case, which is not mandatory to write and ultimately has no effects. Additionally the compiler won’t warn us about this.

Luckily, there is a feature in Swift that I came across a while ago which make expressions like this more clear. You can name your loops with loopName:and when you write break you can specify which loop you want to break. So, if we want to break from the loop in the previous code we can write it like;

gameloop: while true {
  switch gameState {
  case .terminated:
    break gameloop // more clear!
    //...
  }
}

Hope this small tip helps you, especially when you’re debugging your colleague code and want to break free(!) from multiple nested loops that have multiple nested switches and if statements :D.
